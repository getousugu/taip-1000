# TYPING NEXUS — 設計思想・技術ドキュメント

## このソフトについて

AI が生成した日本語長文を課題に、リアルな検定環境でタイピングを測定するアプリ。  
一般的なタイピングゲームとは異なり、**「試験」としての厳密さ**を最優先に設計している。

---

## コンセプト & こだわり

### 試験モードとしての設計

市場の多くのタイピングアプリは「ゲーム」的なアプローチ（スコアのインフレ、即時フィードバック、リトライのしやすさ）を取る。  
このソフトは**日本語タイピング検定**（ビジネス能力検定タイピングなど）の実技に近い形式を目指した。

- 入力中は正誤を一切表示しない（リアルタイムハイライトなし）
- 課題文は試験開始まで確認できず、試験中は課題文を見ながら入力するのみ
- 終了後に初めて採点結果と差分が見える

これは「本番の緊張感に慣れる」という練習目的に最も合った形だと判断した。

---

### AI 課題文生成

毎回 AI（Groq API）が課題文を動的生成する。これには明確な理由がある。

**固定テキストの問題点:**
- 繰り返しで文章を暗記してしまい「タイピング速度」ではなく「記憶力」の測定になる
- 難易度・ジャンルの多様性をプリセットで用意するとメンテナンスコストが高い

**AI 生成の利点:**
- 毎回異なる文章なので暗記が無効化される
- ジャンルを自由入力できるため、業務に近い語彙（IT、医療、法律など）で練習できる
- 難易度の「初級 / 中級 / 上級」を自然言語でプロンプトに組み込むことで、漢字密度・語彙レベルが自動調整される

**Groq を使う理由:**
- 推論速度が圧倒的に速く、1000文字の課題文を数秒で生成できる
- 無料枠が太く、個人ユーザーが API キーを取得してすぐ使い始められる
- `groq-sdk` がブラウザから直接呼び出せる（`dangerouslyAllowBrowser: true`）

APIキーはユーザー自身のものを使うため、サーバーサイドを一切持たない構成にできる。

### LCS（最長共通部分列）による採点

採点の核心は **LCS アルゴリズム**（Longest Common Subsequence）を使ったdiff計算。

```
課題文: 今日は良い天気です
入 力: 今日はいい天気で
```

単純な位置ベースの比較（`target[i] === input[i]`）では、1文字ずれた時点で連鎖的に全てミスになる。  
LCS は文字の「順序的な一致」を求めるため、脱字があっても後続の正解文字が正しくカウントされる。

バックトラック結果を 3 種類に分類する:

| 種別 | 定義 |
|------|------|
| **誤字** | 置き換わった文字（LCSで隣接した脱字と余過を結合したもの） |
| **脱字** | 課題文にあるが入力されなかった文字 |
| **余過** | 課題文にないが余計に入力された文字 |

採点後の**差分ハイライト**はこのバックトラック配列を後処理し、誤字（紫）、脱字（赤）、余過（黄）として HTML に変換して表示する。これにより、成績パネルの数字と画面の表示が1対1で対応する。

**LCSアルゴリズムの弱点克服（独自拡張）:**  
標準的なLCSには、長い課題文の途中で入力が終了した場合、「無関係な後方の文章と散発的にマッチしてしまい、不自然な巨大な差分ブロックができる」というタイピング特有の弱点がある。これを防ぐため以下の2点の改良を施している。
1. **検索ウィンドウの制限:** DPテーブルの計算対象（課題文）を「入力文字数 ＋ 100文字」までに制限。これにより計算量とメモリ使用量を劇的に削減しつつ、後方の無関係な文字を拾い集める暴走を完全に封じている。
2. **前方マッチ優先のバックトラック:** DPテーブルを後ろから計算し、先頭からバックトラックを行うことで、同一文字が複数存在する場合に「最も前にある文字」と優先的に結合させる。これによりタイピングの入力順（左から右）と視覚的な差分が完全に一致する。

**メモリ安全対策:**  
LCS の DP テーブルは `n × m` の `Int32Array` を確保する。  
上記の「検索ウィンドウの制限」により、入力が何文字であっても `(m + 100) × m` に抑えられるため、メモリオーバーフローのリスクは実質的に排除されている（安全装置として `MAX_SAFE_CELLS = 4,000,000` のフォールバックも維持）。

### タイマーの精度

`setInterval` の 1 秒カウントダウンには根本的な問題がある。

- タブが非アクティブのとき、ブラウザはタイマーを意図的に間引く（Throttling）
- CPU 負荷が高いとコールバックが遅延する

このソフトでは `Date.now()` を開始時刻として記録し、毎回インターバルで**実経過秒数**を計算する方式を採用した。インターバルは 500ms にすることで、どんな状況でも表示の最大ズレは 500ms 以内に収まる。

```js
// startTime を記録
this.state.startTime = Date.now();

// 毎 500ms: 実経過から残り時間を計算
const elapsed = Math.floor((Date.now() - this.state.startTime) / 1000);
const remaining = this.state.initialTime - elapsed;
```

### KPM（純速度）の定義

KPM = **K**ey strokes **P**er **M**inute を、このソフトでは次のように定義する。

```
純速度 KPM = 正解文字数 ÷ 経過時間（分）
```

`入力文字数 ÷ 経過時間` にすると、誤字を大量に打った人が高スコアになる矛盾が生じる。  
検定試験の「純速度」の定義に合わせ、**正しく入力できた文字数**を基準にした。

---

### ランク判定

| ランク | 条件 |
|--------|------|
| **S** (金) | KPM > 150 かつ 正確率 > 95% |
| **A** (緑) | KPM > 100 かつ 正確率 > 90% |
| **B** (青) | KPM > 60 かつ 正確率 > 80% |
| **C** (黄) | KPM > 30 |
| **D** (灰) | それ以下 |

正確率は `正解文字数 ÷ 課題文の文字数` で計算する。  
（`÷ 入力文字数` にすると、課題文を無視して短く打ち切った場合に高精度になってしまう）

## 技術スタック

| 要素 | 技術 | 選定理由 |
|------|------|---------|
| フレームワーク | **Vite + Vanilla JS** | シンプルな SPA に余計な抽象レイヤーは不要 |
| AI 推論 | **Groq API** (`groq-sdk`) | 速度・無料枠・ブラウザ対応 |
| 採点 | **自前 LCS 実装** | 外部ライブラリへの依存を避け、アルゴリズムを完全に把握するため |
| フォント | **Noto Sans JP / Inter / Outfit** | 日本語と英数字で最適なフォントを使い分け |
| バックエンド | **なし** | ユーザーが自分の API キーを使うため不要 |

## UI / デザインのこだわり

### グリッチエフェクト
タイトル「TYPING NEXUS」には CSS の `::before` / `::after` 疑似要素を使ったグリッチアニメーションを実装。赤とシアンの色ずれを clip-path で上下に分割し、不定期に発火させることで電子的な雰囲気を演出。

### 段階的な情報開示
- **メニュー画面**: 設定のみ、結果は見えない
- **試験画面**: 課題文と入力欄のみ。スコアはリアルタイムで表示しない（文字数カウントのみ）
- **結果画面**: 全ての採点データ + 差分ハイライト

### 課題文の自動スクロール
入力文字数 ÷ 課題文字数の比率で、課題文ペインのスクロール位置を自動追従させる。  
1000 文字を打ち進めるにつれて、課題文の表示位置が自然に下へ移動する。

### タイマー警告
残り 60 秒になると、タイマーがパルスアニメーションで点滅する。  
色変更だけでなくスケール変化も加え、視覚的な緊張感を演出。
